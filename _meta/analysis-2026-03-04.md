# Configuration Analysis: Objectives vs. Implementation

**Date:** 2026-03-04
**Scope:** Full cross-reference of `_meta/README.md` objectives, `AGENTS.md` rules, all agent definitions, all skill definitions, root `README.md`, and `tools/`.

---

## Executive Summary

The configuration is well-designed and mostly internally consistent. The core objectives — structured quality gates, least-privilege access, specialisation, skill/agent separation, provider-agnostic infra, and mandatory docs/logging — are all clearly reflected in the agent and skill files. However, there are **7 issues** ranging from a workflow gap that undermines a quality gate to documentation drift that could confuse future contributors.

**Critical (blocks a stated objective):** 1
**Major (contradicts a stated rule or creates ambiguity):** 3
**Minor (documentation drift, no runtime impact):** 3

## Issues

### 1. [CRITICAL] `backend-engineer.md` does not invoke `observability-reviewer`

**Objective violated:** Definition of Done item 5 in `AGENTS.md` requires all three reviewers (code, security, observability) to pass before a task is done.

**What's wrong:** `backend-engineer.md` workflow (steps 7–11) only invokes `code-reviewer` and `security-reviewer`. There is no step for `observability-reviewer`. By contrast, `frontend-engineer.md` correctly includes it at steps 11–12.

**Impact:** Backend tasks will routinely skip the observability quality gate. This directly undermines the decision documented in `_meta/decisions.md` ("both backend-engineer and frontend-engineer must invoke it after security-reviewer passes").

**Fix:** Add steps 11–12 to `backend-engineer.md` mirroring `frontend-engineer.md`:

```
11. Once security-reviewer passes, invoke `observability-reviewer` with the same files
12. If `observability-reviewer` returns `"fail"`, address all issues, then re-invoke all three reviewers from step 7
```

Renumber step 11 (report back to build) to step 13.

### 2. [MAJOR] `build.md` quality gate checklist says "both reviewers" — should say "all three reviewers"

**Objective violated:** Consistency with AGENTS.md and the observability-reviewer decision.

**What's wrong:** `build.md` quality gate #2 reads: "Each engineer reports both reviewers passed (no critical or major issues)." Since the addition of observability-reviewer, this should read "all three reviewers."

**Also:** The agent delegation summary table in `build.md` lists `code-reviewer` and `security-reviewer` but omits `observability-reviewer`.

**Fix:** Update quality gate #2 to reference three reviewers. Add `observability-reviewer` to the delegation summary table (even if build doesn't invoke it directly, the table documents the full roster).

---

### 3. [MAJOR] `code-reviewer.md` verdict logic is inconsistent with AGENTS.md gate

**Objective violated:** Structured quality gates with clear pass/fail semantics.

**What's wrong:** `code-reviewer.md` defines its verdict as:
- `"fail"` if any **critical** issues exist
- `"pass_with_issues"` if only **major** or **minor** issues exist
- `"pass"` if none

But `AGENTS.md` says a task cannot proceed with `critical` or `major` issues from **any** reviewer. The `security-reviewer` and `observability-reviewer` correctly define `"fail"` as any `critical` **or** `major` issue. The `code-reviewer` treats `major` as `"pass_with_issues"`.

This means `code-reviewer` can return `"pass_with_issues"` with major issues, and an engineer reading the verdict literally might think it's acceptable. The AGENTS.md gate catches it, but the inconsistency creates confusion about what the verdict means.

**Fix:** Align `code-reviewer.md` verdict logic with the other two reviewers:
- `"fail"` if any `critical` **or** `major` issues exist
- `"pass_with_issues"` if only `minor` issues exist
- `"pass"` if none

### 4. [MAJOR] `qa.md` contract does not mention `observability-reviewer` as a prerequisite

**Objective violated:** Accurate description of the workflow.

**What's wrong:** `qa.md` agent contract says it is invoked "after all engineers report success and code-reviewer/security-reviewer have passed." This omits `observability-reviewer`, which is now a required gate that must pass before QA runs.

**Fix:** Update the contract to read: "after all engineers report success and all three reviewers (code-reviewer, security-reviewer, observability-reviewer) have passed."

### 5. [MINOR] Root `README.md` is significantly out of date

**What's wrong:** The repository structure tree and agent descriptions in the root `README.md` still reflect the pre-split state:
- Lists `agents/engineer.md` (doesn't exist) instead of `backend-engineer.md`, `frontend-engineer.md`
- Missing: `devops-engineer.md`, `developer-advocate.md`, `observability-reviewer.md`
- Agent section describes a single "Engineer" agent, not the split backend/frontend model
- Workflow diagram omits devops-engineer, developer-advocate, observability-reviewer, and logger steps
- Skill roster omits `observability`, `dockerfile-best-practices`, `deployment-planning`, `kubernetes-manifests`

The `_meta/README.md` is the accurate source, and the root `README.md` is explicitly noted as potentially lagging. But the gap is now large enough that a new contributor relying on the root README would get a materially wrong picture of the system.

**Fix:** Regenerate the root `README.md` to match the current state. `@developer-advocate` would be the appropriate agent, but since this is the config repo (not a project repo), a manual update is warranted.

### 6. [MINOR] `logger.md` write permissions may be ambiguous

**What's wrong:** `logger.md` has `permission: edit: deny` and restricted bash access (only `date`, `mkdir`, `cp`). It does not have an explicit `tools:` section setting `write: true`. The logger needs to write the task log file (`agent-logs/{timestamp}/{task-name}.md`).

If OpenCode's default for `write` when no `tools:` key is present is `true`, this works. If the default is `false` or inherits the `edit: deny` setting, the logger cannot create the task log.

**Risk:** Depends on OpenCode's default tool permissions. If defaults are permissive, no issue. If not, the logger silently fails to write logs.

**Fix:** Explicitly add `tools: write: true` to `logger.md` frontmatter to remove ambiguity.

### 7. [MINOR] `_meta/decisions.md` still shows `[IN PROGRESS]` tag on a resolved decision

**What's wrong:** The observability reviewer decision heading reads `## [IN PROGRESS] Observability reviewer agent` but the status line says `Resolved — implemented`. The `[IN PROGRESS]` tag in the heading is stale.

**Fix:** Remove `[IN PROGRESS]` from the heading.

## Objectives Assessment

| Objective (from `_meta/README.md`) | Met? | Notes |
|---|---|---|
| Structured quality gates, not ad-hoc checks | **Partial** | All three reviewers use structured JSON verdicts. But code-reviewer's verdict threshold is inconsistent with AGENTS.md (issue #3), and backend-engineer skips observability-reviewer entirely (issue #1). |
| Principle of least access | **Yes** | All reviewer subagents are read-only with no write or bash access. Architect is read-only. Build has no edit access. DevOps has scoped bash. Logger has minimal bash. Well-enforced. |

| Skills encode conventions; agents encode workflow | **Yes** | Clean separation. Observability skill covers stack-specific conventions; the agent covers the review workflow. TDD skill covers discipline; engineers encode when and how to apply it. |
| Provider-agnostic infrastructure | **Yes** | DevOps agent explicitly prioritises Docker as portability layer, avoids vendor lock-in, confirms before K8s. Deployment-planning and dockerfile skills reinforce this. |
| Every task ends with docs and a log | **Yes** | Developer-advocate and logger are mandatory in the workflow. Build's orchestration protocol lists them as required steps. Definition of done items 9–11 enforce this. |

## Recommendations (Priority Order)

1. **Fix backend-engineer.md** — Add observability-reviewer invocation. This is the most impactful issue; without it, the observability quality gate is only enforced on frontend work.

2. **Align code-reviewer verdict logic** — Match security-reviewer and observability-reviewer: `"fail"` on critical or major.

3. **Update build.md** — Change "both reviewers" to "all three reviewers" and add observability-reviewer to the delegation table.

4. **Update qa.md contract** — Include observability-reviewer in the prerequisite list.

5. **Update root README.md** — Bring it in line with the actual repo state.

6. **Clarify logger write permissions** — Add explicit `tools: write: true`.

7. **Clean up decisions.md heading** — Remove stale `[IN PROGRESS]` tag.
